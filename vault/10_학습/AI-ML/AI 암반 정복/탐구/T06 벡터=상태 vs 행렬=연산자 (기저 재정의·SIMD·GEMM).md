# 벡터([ˈvɛktər], 벡터, 벡터)=상태 vs 행렬(matrix([ˈmeɪtrɪks], 메이트릭스, 매트릭스))=연산자 — 기저 재정의·SIMD([sɪmd], 심드, 심드)·GEMM([dʒɛm], 젬, 젬)

> "벡터는 1차원 배열, 행렬은 그걸 쌓은 2차원 배열" — 대다수가 갖는 *데이터 구조* 관점은 **메모리 표현일 뿐 수학적 본질이 아니다**. 본질에선 **벡터 = 공간 속 한 객체(상태(State([steɪt], 스테이트, 스테이트)))**, **행렬 = 공간 자체를 변형하는 연산자(선형 사상(Linear Mapping([ˈlɪniər ˈmæpɪŋ], 리니어 매핑, 리니어 매핑)))**다. 이 한 끗이 기하(점 vs 격자 재정의)와 하드웨어(대역폭 한계 vs 연산 한계)까지 갈라놓는다. 행렬 단원에 들어가기 전에 *왜 행렬이 "곱하는 쪽"인가*를 못 박는다.

> 🌱 **탐구 노트(재사용)** — 책 절이 아니라 호기심으로 파고든 조사. 행렬을 다루는 어느 노트에서든 링크로 재사용.
> 🔗 유래·적용: [[04 벡터 내적 — 점곱·기하·직교 + 아다마르곱·외적|§1.5 외적=rank-1 행렬]] · [[T04 NumPy 벡터 내부 — shape·strides·열벡터 (N,1)|T04 ndarray=1D블록+메타]]
> ⚙️ 원리 이웃: [[GPU 하드웨어 구조 — SIMT·SM·텐서코어 (ARM 관점)|GPU 하드웨어(SIMT·GEMM)]] · [[3D 변환 행렬 — 동차좌표·회전·투영 (선형대수 응용)|3D 변환 행렬(선형변환)]] · 💡 [[왜 선형대수인가 — 데이터·알고리즘의 표준 언어|왜 선형대수인가]] ｜ 🗂️ [[_탐구 MOC]] · 🌐 [[_지식 연결 허브]]

> 🎯 **핵심 (TL;DR)**
> - **수학적 본질:** 벡터=**상태**(공간 속 한 점/객체) ↔ 행렬=**선형 사상(연산자(operator([ˈɒpəreɪtər], 오퍼레이터, 오퍼레이터)))**. `Av`는 변수 `x`를 함수 `f(x)`에 넣는 것과 같다 — **행렬은 입력 벡터를 출력 벡터로 바꾸는 '함수' 그 자체.**
> - **기하:** 벡터=주어진 기저(basis([ˈbeɪsɪs], 베이시스, 베이시스)) 위 화살표 끝점. 행렬=**기저 자체를 갈아엎음** — *행렬 A의 각 열(column) = 변환 후 기저축이 도착하는 위치*. 그래서 곱은 공간을 회전·전단·신축한다.
> - **하드웨어:** 벡터 연산(원소별)은 **메모리 대역폭(bandwidth([ˈbændwɪdθ], 밴드위드, 밴드위드)) 한계**, 행렬곱(GEMM)은 **연산(compute) 한계**. 가르는 진짜 축은 SIMD냐 GEMM이냐가 아니라 **산술 밀도(arithmetic intensity([əˈrɪθmətɪk ɪnˈtɛnsəti], 어리스메틱 인텐시티, 어리스메틱 인텐시티), FLOP/byte)**.
> - **단, 과교정 주의:** "행렬은 데이터 표가 아니다"는 **반만 맞다** — ML에서 가중치 행렬 W는 *연산자*지만, 데이터 행렬 X·공분산·이미지·인접행렬은 엄연히 *데이터 컨테이너*다. **행렬은 두 역할을 겸한다.**

> [!note] 🗺️ 이 노트 읽는 법
> Gemini 탐구를 1차 출처로 **교차검증·교정**해 정리했다. 결론은 §1, 본질(WHY)은 §2~4. 하드웨어 세부(SIMT·텐서코어)는 중복을 피해 [[GPU 하드웨어 구조 — SIMT·SM·텐서코어 (ARM 관점)|GPU 노트]]로 위임하고, 여기선 *벡터/행렬을 가르는 축*만 본다.

## 1. 한 장 요약 — 세 층위에서 갈린다
| 층위 | 벡터(Vector) | 행렬(Matrix) |
|---|---|---|
| **수학적 본질** | 선형 공간의 한 원소 = **상태(State)** | 공간을 바꾸는 **선형 사상 = 연산자(함수)** |
| **기하** | 기저 위 화살표의 끝점(고정 위치) | **기저(축) 자체의 재정의** — 공간을 비틀·돌림 |
| **하드웨어** | 원소별 연산 → **대역폭 한계**(낮은 산술밀도) | 데이터 재사용 행렬곱(GEMM) → **연산 한계**(높은 산술밀도) |

핵심 한 줄: **벡터는 '연산을 당하는 대상', 행렬은 '연산을 하는 주체'.** `1차원·2차원 배열`은 이 본질을 담는 *메모리 그릇*일 뿐이다(→ [[T04 NumPy 벡터 내부 — shape·strides·열벡터 (N,1)|ndarray는 1D블록+메타]]).

## 2. 수학적 본질 — 상태(State) vs 연산자(Operator)
- **벡터 = 상태.** 가감산·스칼라곱이 정의되는 선형 공간(Linear Space)의 한 원소. 데이터 분석에선 *하나의 관측치*, 임베딩 공간에선 *개념의 좌표(특징값)* 라는 **고정된 '값'**.
- **행렬 = 선형 사상(연산자).** 본질은 데이터 표가 아니라 *공간을 왜곡·회전·신축하는 동적 연산자*. 벡터 $v$ 에 행렬 $A$ 를 곱하는 행위 $Av$ 는 변수 $x$ 를 함수 $f(x)$ 에 대입하는 것과 **수리적으로 동일**하다.

$$
A v = w \quad\Longleftrightarrow\quad f(x)=y \qquad (\text{행렬} = \text{선형 함수})
$$

> [!warning] ⚠️ 정확성 — "행렬은 데이터 표가 아니라 연산자다"는 **과교정**   `#정확성`
> Gemini([ˈdʒɛmɪnaɪ], 제미나이, 제미니)는 "행렬의 본질은 데이터 표가 아니라 연산자"라고 단정했는데, **반만 맞다.** 순수 선형대수의 *깊은 본질*은 선형 사상이 맞지만, ML 실무에서 행렬은 **두 역할을 겸한다**:
> - **연산자 역할:** 가중치 행렬 $W$ ($Wx{+}b$), 회전/투영 행렬 — *공간을 바꾼다*.
> - **데이터 컨테이너 역할:** 데이터 행렬 $X$ `(n_samples, n_features)`, 공분산 행렬, 이미지(픽셀 격자), 인접 행렬 — *값을 담는다*.
>
> "2차원 배열" 직관을 **버리라**는 게 아니라, 거기에 **'연산자'라는 또 하나의 정체를 더하라**가 정확하다. 무엇이 무엇에 곱해지느냐(연산자 ↔ 피연산자)는 *문맥*이 정한다.

## 3. 기하 — 점/화살표 vs 기저(공간) 재정의
- **벡터:** 주어진 기저 시스템 위에서 화살표의 끝점을 가리킨다. 기저가 안 바뀌는 한 *고정된 위치 벡터*.
- **행렬:** **새 기저 벡터들의 집합.** 행렬 $A$ 의 각 **열(column)** 은 변환 후 기저축 $\hat{\imath},\hat{\jmath}$ 가 *어디로 가는지*를 적어둔 것. 그래서 행렬 곱은 기존 직교 좌표계를 찌그러뜨리거나 회전시켜 **공간 자체를 재정의**한다.

🔄 재구성 그림 — 같은 벡터 v가, 행렬을 만나면 격자째 끌려간다(우: 열 = 기저가 도착하는 곳).

![[T06-1 state-vs-operator.png|640]]

> [!tip] 🧭 외적과의 연결 — 행렬은 "어디서 오나"   `#연결`
> [[04 벡터 내적 — 점곱·기하·직교 + 아다마르곱·외적|§1.5]]에서 본 **외적 $vw^\mathsf{T}$ = rank-1 행렬**이 바로 "벡터로 빚은 가장 단순한 연산자"다. 일반 행렬 = rank-1 연산자들의 합(→ SVD([ɛs viː diː], 에스브이디, 에스브이디)·LoRA([ˈloʊrɑː], 로라, 로라)의 토대). 즉 *벡터(상태) → 외적 → 행렬(연산자)* 로 층이 올라간다.

> [!example] 🧩 책 코드 — [필수수학] §4.2 행렬-벡터 곱 = 기저 행렬 적용   `#연결/코드`
> 필수수학 §4.2는 이 "열 = 기저의 상" 개념을 코드로 못 박는다. **기저 벡터 î, ĵ 를 *열*로 쌓아 행렬을 만들고**(넘파이 `array([i_hat,j_hat])`는 *행*으로 채우므로 **전치** 필요 → [[03 벡터 연산 — 덧셈·뺄셈·스케일링·전치·브로드캐스팅|전치]]) `basis.dot(v)` 로 변환한다:
> ```python
> import numpy as np
> # 예제 4-9: î=[2,0], ĵ=[0,3] → 공간을 2·3배 신축
> i_hat, j_hat = np.array([2,0]), np.array([0,3])
> basis = np.array([i_hat, j_hat]).T        # 열로 세움(전치)
> print(basis.dot(np.array([1,1])))          # [2 3]
> # 예제 4-10: î=[2,3], ĵ=[2,-1] → 전단+회전+반전 복합
> basis = np.array([[2,3],[2,-1]]).T
> print(basis.dot(np.array([2,1])))          # [6 5]
> ```
> 핵심: $\begin{bmatrix}x_{new}\\y_{new}\end{bmatrix}=\begin{bmatrix}a&b\\c&d\end{bmatrix}\begin{bmatrix}x\\y\end{bmatrix}$ — *행렬 = 기저 벡터로 표현된 하나의 변환*. 4종 변환(스케일·회전·전단·반전) 기하는 [[3D 변환 행렬 — 동차좌표·회전·투영 (선형대수 응용)|3D 변환 노트]]. 기저 개념은 [[06 벡터 확장 — 집합·선형결합·선형독립·부분공간·생성·기저|06]].

## 4. 하드웨어 — SIMD vs GEMM = 대역폭 한계 vs 연산 한계
컴퓨터 아키텍처·커널(CUDA([ˈkuːdə], 쿠다, 쿠다)/Triton([ˈtraɪtɒn], 트라이튼, 트라이튼)) 레벨에서 둘을 다루는 명령어와 **병목 지점**이 다르다.

- **벡터 연산(원소별 덧셈·스칼라곱):** CPU의 **AVX([eɪ viː ɛks], 에이브이엑스, 에이브이엑스)(SIMD)**·GPU의 **SIMT([sɪmt], 심트, 심트)** 로 수행되는 단순 병렬 연산. 데이터를 한 번 읽어 1~2회 연산하고 버린다 → **메모리 대역폭 한계(Memory-Bandwidth Bound)**. 산술 밀도가 극도로 낮다.
- **행렬곱(GEMM, General Matrix Multiply):** 엔비디아 **텐서 코어(Tensor Core([ˈtɛnsər kɔːr], 텐서 코어, 텐서 코어))**·구글 **TPU([tiː piː juː], 티피유, 티피유)** 의 **MXU([ɛm ɛks juː], 엠엑스유, 엠엑스유)(시스톨릭 어레이(systolic array([sɪˈstɒlɪk əˈreɪ], 시스톨릭 어레이, 시스톨릭 어레이)))** 가 처리. 같은 데이터를 캐시에 두고 $n^3$ 번 재사용 → **연산 한계(Compute Bound)**. 산술 밀도가 높아 하드웨어 연산기를 꽉 채운다.

🔄 재구성 그림 — 산술 밀도(FLOP/byte)가 병목을 가른다. 벡터 연산은 대역폭에서 굶고, GEMM은 연산기를 포화시킨다.

![[T06-2 arithmetic-intensity.png|620]]

> [!warning] ⚠️ 정확성 — "벡터=SIMD / 행렬=GEMM" 이분법의 한계   `#정확성`
> Gemini의 분류엔 두 가지 부정확이 섞여 있다:
> - **① SIMD ≠ GPU 일반 코어.** Gemini는 "GPU의 일반 코어에서 SIMD"라 했지만, 정확히는 **CPU = SIMD(AVX)**, **GPU = SIMT**(스레드 단위 단일 명령)다(→ [[GPU 하드웨어 구조 — SIMT·SM·텐서코어 (ARM 관점)|GPU 노트]]).
> - **② GEMM도 바닥에선 SIMD/SIMT 위에 올라간다.** "벡터=SIMD vs 행렬=GEMM"은 *층위가 다른 것을 나란히 놓은* 비유다. 둘을 진짜 가르는 축은 명령어 종류가 아니라 **산술 밀도(arithmetic intensity)** — 즉 **루프라인(Roofline([ˈruːflaɪn], 루프라인, 루프라인)) 모델**의 대역폭 한계 vs 연산 한계다. 시스톨릭 어레이도 *단일 사이클*이 아니라 데이터를 흘려보내며 **파이프라인(pipeline([ˈpaɪplaɪn], 파이프라인, 파이프라인))** 으로 MAC을 누적한다(높은 재사용이 핵심).
>
> *교정 요지:* 표현(SIMD/GEMM)이 아니라 **데이터 재사용율**이 본질. 행렬곱이 GPU의 총아인 이유는 "행렬이라서"가 아니라 **산술 밀도가 높아서**다.

> [!quote] 🌉 다리 — 하드웨어가 딥러닝을 "행렬곱으로" 강제했다   `#연결/CS`
> 낮은 산술밀도(벡터·분기)는 GPU에서 손해 → 딥러닝은 *분기 없는 계층적 행렬곱(GEMM)* 형태로 **강제 진화**했다. 같은 결론을 하드웨어 쪽에서 본 게 [[GPU 하드웨어 구조 — SIMT·SM·텐서코어 (ARM 관점)|GPU 노트]]의 '분기의 저주·공생'. *왜 모델이 행렬 천국인가 = 실리콘이 그렇게 깎였기 때문.*

### 4.1 Av의 두 순회 패턴 — 행별 내적 vs 열별 누적(gaxpy)
Av를 계산기가 실제로 어떻게 훑는지도 §2~3의 두 시점과 그대로 대응한다 — **다른 개념이 아니라 같은 두 시점을 계산 루프로 옮긴 것.**

- **행별(내적) 순회:** 출력 원소 하나씩 $y_i=(\text{row}_i A)\cdot v$ — [[2.2 행렬 곱셈·행렬식|원자 2.2]]의 "칸=내적" 시점 그대로.
- **열별(누적) 순회 = gaxpy(General A times X Plus Y, [ˈɡæks.pi], 객스피, 객스피):** $y\leftarrow 0$ 에서 시작해 열마다 $y \mathrel{+}= v_j\cdot(\text{col}_j A)$ 로 누적 — 원자 2.2의 "열=선형결합" 시점 그대로. 열-우선(column-major, Fortran/BLAS 관행) 저장에서 각 열이 메모리에 연속이라 캐시 효율이 좋아, 고전 BLAS([blæs], 블래스, 블라스) Level-2(행렬-벡터) 표준 알고리즘으로 채택됨.

> [!warning] ⚠️ 정확성 — "GPU가 이렇게 계산한다"는 과단순화   `#정확성`
> gaxpy는 **CPU·고전 BLAS의 계산 순서**로선 정확하지만, 이걸 그대로 **GPU 텐서 코어의 물리적 실행 방식**이라 하면 부정확하다. 텐서 코어는 행/열을 순차로 훑지 않고 **타일(tile) 단위 블록을 공유 메모리에 올려 병렬 FMA**로 처리한다(위 §4 SIMD/SIMT 정확성 지적과 같은 결). "열 시점이 계산 순서로도 유효하다"는 **수학적 계산 그래프** 얘기지, 실리콘 위 물리적 순서로 일반화하면 틀린다.

### 4.2 배치(Batch) — 왜 실무에선 Av가 아니라 AX인가
> 🌱 계기: [[벡터와 행렬의 관계 (mathlab수학력발전소 영상 정리)|영상]]의 머신러닝 장면(입력 벡터→가중치 행렬→출력 벡터, 06:47) — "입력이 왜 벡터 하나가 아니라 행렬(배치) 형태로 들어오나"를 파고든 결과.

**입력이 여러 개면 그냥 나란히 쌓아도 함수 정체성은 안 깨진다.** 선형성(분배법칙) 덕분에:
$$
AX = A[\,x_1 \; x_2 \; \cdots \; x_k\,] = [\,Ax_1 \; Ax_2 \; \cdots \; Ax_k\,]
$$
즉 $k$개 벡터를 행렬 $X$로 묶어 한 번에 곱해도, 결과는 *각 벡터를 따로 변환한 것을 다시 옆으로 붙인 것*과 수리적으로 100% 동일하다 — [[2.2 행렬 곱셈·행렬식|원자 2.2]]의 "열=선형결합" 시점 그대로.

**그런데도 실무는 굳이 배치로 묶는다 — 이유는 수학이 아니라 하드웨어.**
- 벡터 하나만 넣어 $Av$(GEMV, General Matrix-**V**ector, BLAS Level-2)를 구하면, 가중치 $A$를 VRAM에서 캐시로 읽어오는 시간(메모리 대역폭 비용) 동안 GPU 코어 대부분이 논다 — §4.1의 gaxpy가 바로 이 저효율 패턴.
- $k$개를 $X$로 묶어 $AX$(GEMM, BLAS Level-3)를 구하면, 같은 $A$를 **한 번 읽어 $k$번 재사용** — 산술 밀도가 올라가 텐서 코어가 꽉 찬다(§4의 arithmetic intensity 논리 그대로).
- ⇒ **입력을 행렬로 넣는 이유는 함수의 본질이 바뀌어서가 아니라, GEMV(메모리 대역폭 한계)를 GEMM(연산 한계)으로 바꿔 하드웨어를 낭비하지 않기 위해서다.**

**행렬 놈이 여기서 하는 일 — 배치 전체의 안전선.** [[2.5 행렬 노름·대각합 — 프로베니우스·ℓ2·tr(AᵀA)|원자 2.5]]의 유도된 놈 정의 $\lVert A\rVert=\max_{x\neq0}\lVert Ax\rVert/\lVert x\rVert$ 는 곧바로 부등식 $\lVert Ax_i\rVert \le \lVert A\rVert\,\lVert x_i\rVert$ 를 준다. 배치 $X$의 벡터가 1개든 1만 개든, **출력 $AX$의 어떤 열도 "입력 크기 × ‖A‖"를 못 넘는다** — 유도된 놈이 배치 크기와 무관하게 성립하는 시스템 상한선인 이유.

## 🧠 통찰 — 한 줄
**벡터는 명사(상태), 행렬은 동사(연산자).** "1D/2D 배열"은 그 명사·동사를 담는 *문법 그릇*일 뿐이고, 실리콘은 동사(행렬곱)를 *재사용이 많다는 이유로* 편애한다. 그래서 "데이터를 어떻게 배열하나"가 아니라 **"무엇이 무엇을 변환하나(연산자↔상태)"** 로 보면 수학·기하·하드웨어가 한 줄로 꿰인다.

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **선형 사상 / 연산자 (Linear Mapping / Operator)** | 벡터를 벡터로 보내는 선형 함수 = 행렬의 본질; $Av$ = $f(x)$ |
| **상태 vs 함수 (State vs Function)** | 벡터=공간 속 고정된 값(상태) ↔ 행렬=값을 바꾸는 규칙(함수) |
| **기저 재정의 (열 = 기저의 상)** | 행렬의 각 열 = 변환 후 기저축 $\hat{\imath},\hat{\jmath}$ 의 도착 위치 |
| **산술 밀도 (Arithmetic Intensity)** | FLOP/byte; 연산량 ÷ 메모리 이동량 — 병목을 가르는 지표 |
| **대역폭 한계 / 연산 한계 (Bandwidth- / Compute-Bound)** | 메모리 전송이 병목(벡터 연산) / 연산기가 병목(GEMM) |
| **SIMT** | Single Instruction Multiple Threads; GPU 병렬 모델(CPU의 SIMD에 대응) |
| **시스톨릭 어레이 (Systolic Array)** | 데이터를 흘려보내며 MAC을 파이프라인 누적하는 행렬곱 전용 격자(TPU MXU) |
| **gaxpy** | 열별 누적으로 Av 계산($y{+}{=}v_j{\cdot}\text{col}_jA$); 열-우선 저장에서 캐시 효율적인 고전 BLAS Level-2 알고리즘 |
| **GEMV (General Matrix-Vector multiply)** | Av 연산의 BLAS 이름(Level-2); 메모리 대역폭 한계 — 배치 없이는 GPU가 논다 |
| **배치(Batch) = AX** | 벡터 k개를 행렬로 묶어 한 번에 곱함; 함수 정체성은 안 바뀜(Ax₁,…,Axₖ와 동일), GEMV→GEMM(Level-3) 전환이 목적 |
