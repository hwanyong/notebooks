# NumPy([ˈnʌmpaɪ], 넘파이, 넘파이) 벡터([ˈvɛktər], 벡터, 벡터) 내부 — shape([ʃeɪp], 셰이프, 셰이프)·strides([straɪdz], 스트라이즈, 스트라이즈)·열벡터 (N,1)

> 왜 [실전선대]는 열벡터를 `np.array([[1],[2],[3]])`(2D)로 쓰고 [필수수학]은 `np.array([1,2,3])`(1D)로 쓸까? 근저엔 **NumPy ndarray([ɛndiːəreɪ], 엔디어레이, 엔디어레이) = 1D 연속 메모리 블록 + (shape·strides) 메타데이터**라는 정체가 있다(= C의 `arr[row*COLS+col]` 포인터 산술을 NumPy가 대신 해주는 것). 이 차이가 `.T`·`@`·`np.linalg`에서 어떻게 갈리는지, 그래서 노트엔 왜 **명시적 2D (N,1)** 로 통일하는지 정리한다.

> 🌱 **탐구 노트(재사용)** — 책 절이 아니라 호기심으로 파고든 조사. 어느 벡터 노트에서든 링크로 재사용.
> 🔗 유래·적용: [[01 벡터 — 정의·차원·방향 (NumPy 생성·시각화)|§1.1 벡터 정의]] · [[03 벡터 연산 — 덧셈·뺄셈·스케일링·전치·브로드캐스팅|필수수학 §4.1]] · [[03 벡터 연산 — 덧셈·뺄셈·스케일링·전치·브로드캐스팅|§1.2 전치·브로드캐스팅]]
> ⚙️ 원리 이웃: [[메모리·캐시·멀티프로세스 (시스템 원리)|CS 메모리 레이아웃]] · 💡 [[왜 선형대수인가 — 데이터·알고리즘의 표준 언어|왜 선형대수인가]] ｜ 🗂️ [[_탐구 MOC]] · 🌐 [[_지식 연결 허브]]

> 🎯 **핵심 (TL;DR)**
> - **ndarray = 1D 연속 블록 + 메타데이터.** 메모리엔 `[1,2,3,4,5,6]` 한 줄만. `shape`(형태)와 `strides`(다음 칸까지 몇 바이트)가 "2D인 척" 연기. ⇒ **C 포인터 산술 `arr[row*COLS+col]`을 NumPy가 대신.**
> - **세 표현은 다른 객체:** `(N,)` 1D(축 없음) ≠ `(N,1)` 열벡터(2D) ≠ `(1,N)` 행벡터(2D).
> - **`.T`의 함정:** `(N,)`은 전치해도 그대로 `(N,)`(뒤집을 축이 없음). `(N,1)`만 `.T`→`(1,N)`로 제대로 눕는다. ✅ **책 공식 인증(2026-07):** [실전선대] §4.4 각주 3 — "1D 배열에 전치를 적용하면 아무런 일도 일어나지 않는다. 경고나 오류 없이 동일한 1차원 배열 반환" → [[2.3 행렬 전치 — 내적·외적 표기법·LIVE EVIL·대칭 행렬|2.3]].
> - **`@`·`np.linalg`:** `(N,)@(N,)`는 *행렬곱이 아니라 내적*(스칼라). `np.linalg.inv/det/eig`는 **2D를 요구**(1D면 `LinAlgError`).
> - **그래서 노트 규칙:** 수식의 열벡터는 **명시적 `(N,1)`** 로 적어 *수식↔코드 1:1*. (단, 실무 ML([ɛm ɛl], 엠엘, 엠엘)은 1D `(N,)`·배치 2D `(n,feat)`도 흔히 씀 → 둘 다 알기.)

> [!note] 🗺️ 이 노트 읽는 법
> 책 §은 "벡터 선언"만 보여주고 넘어가지만, 두 책 코드가 *왜 다른지*를 NumPy 내부(메모리·메타데이터)로 파고든 탐구. **결론(규칙)은 §5, 이유(WHY)는 §1~4.**

## 1. 두 책의 표현 차이
같은 열벡터 $v=\begin{bmatrix}1\\2\\3\end{bmatrix}$ 을 —

| 책 | 코드 | shape | 정체 |
|---|---|---|---|
| **[실전선대]** (코헨) | `np.array([[1],[2],[3]])` | `(3,1)` | 엄격한 **2D 열행렬**(수학적 자격 보존) |
| **[필수수학]** (닐드) | `np.array([1,2,3])` | `(3,)` | **1D 배열**(축 없음, 평탄) |

- [실전선대]는 *선형대수 규칙을 코드로 증명*이 목적 → 가로(1×3)·세로(3×1)를 **일부러 분리**해 `(3,1)`로 가둠.
- [필수수학]은 *데이터 과학 구현*에 방점 → 굳이 대괄호 두 겹 안 쓰고 `(3,)`로 단순화.

> [!warning] ⚠️ 정확성 — "데이터 과학은 1D로 다룬다"는 부정확   `#정확성`
> Gemini([ˈdʒɛmɪnaɪ], 제미나이, 제미니)는 [필수수학]의 1D를 "데이터 과학(Scikit-learn([ˈsaɪkɪt lɜːrn], 사이킷런, 사이킷런) 등)이 1D로 다뤄서"라 했지만, **sklearn([ɛsˈkɛlɜːrn], 에스케이런, 에스케이런)은 오히려 2D `(n_samples, n_features)`를 요구**하고 1D를 주면 경고/에러를 낸다. [필수수학]의 1D는 *데이터 과학의 표준*이라서가 아니라 **입문 단순화**로 보는 게 정확하다.

## 2. NumPy ndarray의 정체 = 1D 블록 + 메타데이터
`ndarray`는 내부적으로 **연속된 1D 메모리 블록(buffer([ˈbʌfər], 버퍼, 버퍼))** 하나 + 그 위에 얹은 메타데이터다. 컴퓨터는 데이터가 1D인지 2D인지 모른다 — NumPy가 두 메타데이터로 "다차원인 척" 연기한다.

- **`shape`** — 차원의 형태. 예: `(2,3)`.
- **`strides`** — *다음 행/열로 가려면 메모리를 몇 바이트 건너뛰나*. (C의 인덱스 산술을 대신)

```python
m = np.arange(6).reshape(2, 3)
# 내부 1D 버퍼:  [0 1 2 3 4 5]   ← 메모리는 한 줄
m.shape     # (2, 3)
m.strides   # (24, 8)  → 다음 '행'=+24바이트(=3개×int64 8B), 다음 '열'=+8바이트
```

> [!quote] 🌉 다리 — `strides` = C 포인터 산술 `arr[row*COLS+col]`   `#연결/CS`
> C에서 다차원 배열을 1D로 깔고 `base + (row*COLS + col)*itemsize`로 접근하던 그 계산을, NumPy는 **`strides`에 미리 담아 자동 수행**한다. `(2,3)` int64면 `row` 한 칸 = `3*8=24B`, `col` 한 칸 = `8B`. → 즉 *"1차원 평탄 배열 + 인덱스 산술"* 이라는 네 추측이 **NumPy 내부에서 정확히 일어난다.** 전역 다리 → [[_지식 연결 허브]] · 원리 이웃 [[메모리·캐시·멀티프로세스 (시스템 원리)|메모리 레이아웃]]

이래서 **같은 1D 버퍼라도 `shape`/`strides`만 바꾸면** 다른 차원의 배열처럼 동작한다(그래서 `reshape`·전치가 대개 *복사 없이* 메타데이터만 수정 = 빠름).

## 3. `(N,)` vs `(N,1)` vs `(1,N)` — 다른 객체
```python
flat = np.array([1,2,3])        # (3,)   1D, 축(가로/세로) 개념 없음
col  = np.array([[1],[2],[3]])  # (3,1)  2D 열벡터
row  = np.array([[1,2,3]])      # (1,3)  2D 행벡터
```
- `(N,)`은 **방향이 없다**(행도 열도 아님). 수학의 열벡터를 *흉내*낼 뿐.
- `(N,1)`/`(1,N)`만 *진짜* 열/행 — 2개 축을 가져 전치·행렬곱 규격에 들어맞는다.

### `.T` 전치의 함정
```python
flat.T.shape   # (3,)   ← 변화 없음! 뒤집을 축이 없다
col.T.shape    # (1,3)  ← 열 → 행으로 제대로 누움
```
수식의 $v^\mathsf{T}$ 를 코드로 옮길 때 `(N,)`이면 **`.T`가 먹통** → 버그. `(N,1)`이라야 의도대로 작동.

## 4. 1D가 깨지는 두 곳
### ① `@` / `np.dot` — 규격이 다르게 해석됨
```python
flat @ flat        # 14   ← 1D@1D = '내적(스칼라)'. 행렬곱 아님
row @ col          # [[14]] shape (1,1)  ← 진짜 행렬곱
col.T @ col        # [14]  = Σvᵢ²  (vᵀv, 노름의 제곱)
```
> [!warning] ⚠️ 정확성 — `(N,)@(N,)`는 "브로드캐스팅"이 아니다   `#정확성`
> Gemini는 이를 "브로드캐스팅(자동 유추)"이라 했지만, 정확히는 **NumPy의 1D@1D = 내적(inner product([ˈɪnər ˈprɒdʌkt], 이너 프로덕트, 이너 프로덕트)) 규칙**이라 스칼라가 나온다(브로드캐스팅은 *모양이 다른 배열*을 맞춰 원소연산하는 별개 기능). 어느 쪽이든 *"행렬곱 규격(2D)을 의도했는데 다르게 동작"* 한다는 함정은 같다.

### ② `np.linalg` — 2D를 요구
```python
np.linalg.det(np.array([[4,7],[2,6]]))   # 10.0  (2D OK)
np.linalg.det(np.array([1,2,3]))         # LinAlgError: 1-dimensional array given...
```
역행렬·행렬식·고유값분해 등 선형대수 엔진은 **최소 2D**라야 동작. 1D면 "행렬이 아니다"라며 멈춘다.

## 5. 그래서 노트 규칙 — 명시적 (N,1) + 매핑 템플릿
**결론:** 수식의 열벡터는 노트에서 **명시적 2D `(N,1)`** 로 적는다. 이유 — ① 수식 기호 ↔ 코드가 1:1로 맞아 버그 틈이 없음, ② 두 책을 "모든 벡터는 `(N,1)`/`(1,N)`으로 통일"이라는 한 규격으로 재해석, ③ shape가 보여 파이프라인 차원 흐름이 눈에 들어옴([[왜 선형대수인가 — 데이터·알고리즘의 표준 언어|아키텍트 시점]]).

> [!tip] 🧭 매핑 템플릿 — [수학] → [책 1D] → [내 2D]   `#규칙`
> [필수수학]의 1D 코드는 *지우지 말고* 아래처럼 **나란히** 적어 대조한다.
> - **수학:** $v=\begin{bmatrix}1\\2\\3\end{bmatrix}$ (세로 열벡터)
> - **책 1D(shortcut):** `np.array([1,2,3])` → `(3,)` *(.T 먹통)*
> - **내 2D(명시):** `np.array([[1],[2],[3]])` → `(3,1)` *(또는 `np.array([[1,2,3]]).T` / `.reshape(-1,1)`)*

> [!info] 🔭 단, (N,1)이 *항상* 정답은 아니다   `#정확성`
> 명시적 `(N,1)`은 **수식 1:1 학습엔 최적**이지만, 실무 ML 코드는 단일 벡터를 1D `(N,)`로, *여러 샘플*을 배치 2D `(n_samples, n_features)`로 두는 게 더 흔하다(브로드캐스팅·sklearn 규격). → 학습용 규칙으로 (N,1)을 쓰되, **1D·배치2D 관용도 함께** 읽을 것.

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **ndarray** | NumPy 핵심 객체 = 1D 연속 메모리 블록 + (dtype·shape·strides) 메타데이터 |
| **연속 메모리 블록** | 데이터가 메모리에 한 줄로 붙어 저장(contiguous([kənˈtɪɡjuəs], 컨티규어스, 컨티규어스)); 다차원은 메타로 연기 |
| **shape** | 차원의 형태 `(행,열,…)`; `(N,)`1D·`(N,1)`열·`(1,N)`행 |
| **strides** | 다음 축 원소로 가는 바이트 간격; C의 `row*COLS+col` 산술을 대신 |
| **(N,) vs (N,1)** | 축 없는 1D(.T·행렬곱 함정) vs 2D 열벡터(수식 1:1) |
| **np.linalg** | 역행렬·행렬식·고유값 등 선형대수 엔진(입력 2D 요구, 1D→LinAlgError) |
| **reshape(-1,1)** | 1D를 (N,1) 열벡터로 변환(−1=자동 추론) |
