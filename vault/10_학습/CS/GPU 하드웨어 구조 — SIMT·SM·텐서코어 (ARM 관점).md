# GPU([dʒiː piː juː], 지피유, 지피유) 하드웨어 구조 — SIMT([sɪmt], 심트, 심트) · SM([ɛs ɛm], 에스엠, 에스엠) · 텐서코어 (ARM([ɑːrm], 암, 암) 관점)

> "GPU로 전송한다"·"GPU가 빠르다"의 *물리적 토대* — 순수 컴퓨터구조 원리. **CPU([siː piː juː], 시피유, 시피유)(ARMv5 등 폰 노이만/SISD([ɛs aɪ ɛs diː], 에스아이에스디, 에스아이에스디))의 구조를 아는 사람**이 GPU를 '상태 전이'로 이해하도록 정리. *특정 프레임워크와 무관한 범용 하드웨어 지식.*

> 🧩 **CS 노트(원리·WHY/WHAT)** — 어떤 GPU 가속에도 적용되는 토대.

> 🎯 **핵심 (TL;DR)**
> - **설계 철학:** CPU = 제어부·캐시([kæʃ], 캐시, 캐시)에 투자 → **지연(Latency([ˈleɪtənsi], 레이턴시, 레이턴시)) 최소화**(똑똑한 장군). GPU = 제어부를 걷어내고 ALU([eɪ ɛl juː], 에이엘유, 에이엘유)를 수천 개 → **처리량(Throughput([ˈθruːpʊt], 스루풋, 스루풋)) 극대화**(일꾼 군단).
> - **SIMT:** 제어부(스케줄러) 하나가 **같은 명령어**를 수천 ALU(스레드)에 브로드캐스트. ARM의 SISD(제어부 1 : 데이터 1)를 "제어부 1 : 데이터 수천"으로 확장.
> - **계층:** SM(연산 엔진) ⊃ CUDA([ˈkuːdə], 쿠다, 쿠다) 코어(ALU)·텐서 코어(행렬곱 전용) + Register([ˈrɛdʒɪstər], 레지스터, 레지스터)·Shared Memory([ʃɛrd ˈmɛməri], 셰어드 메모리, 셰어드 메모리)(개발자 제어 SRAM([ɛs ræm], 에스램, 에스램)) + VRAM([viː ræm], 브이램, 브이램)(Global([ˈɡloʊbəl], 글로벌, 글로벌)).
> - **분기의 저주:** `if`로 스레드([θrɛd], 스레드, 스레드)가 갈라지면 **분기 발산(Branch Divergence([bræntʃ daɪˈvɜːrdʒəns], 브랜치 다이버전스, 브랜치 다이버전스))** → 순차 실행으로 효율 반토막. 그래서 *분기 없는 행렬 연산*에 최적.
> - **공생:** 이 하드웨어 제약이 딥러닝을 *계층적 행렬곱(GEMM([dʒɛm], 젬, 젬) 형태*로 **강제**했다.

## 1. 설계 철학 — 제어부의 희생, 연산부의 확장
CPU와 GPU는 같은 실리콘 면적을 **정반대로** 배분한 선택이다.

| | CPU (ARMv5 등) | GPU |
|---|---|---|
| 코어(ALU) | 적지만 강력한 대형 코어 | 작고 단순한 코어 **수천 개** |
| 제어부(Control) | 크고 복잡(분기 예측·비순차 실행) | 아주 작고 단순(SIMT) |
| 캐시 | 매우 큼(지연 숨김) | 상대적으로 작음(대역폭 중시) |
| 목적 | **지연(Latency) 최소화** — 순차 | **처리량(Throughput) 극대화** — 병렬 |

- CPU = 복잡한 `if-else`·순차 예측에 최적화된 **'똑똑한 장군'**.
- GPU = `A + B = C` 같은 똑같은 연산을 100만 개의 다른 데이터에 동시 수행하는 **'일꾼 군단'**. 그래서 **GPU는 단독 부팅조차 못 하고**, CPU(장군)가 내리는 커널 호출을 수행하는 **병렬 가속기**로만 존재한다.

## 2. GPU 내부 계층 (하드웨어 블록)
- **SM (Streaming Multiprocessor([ˈstriːmɪŋ ˌmʌltiˈprɒsɛsər], 스트리밍 멀티프로세서, 스트리밍 멀티프로세서)):** GPU의 가장 큰 단위 블록 = 작은 '연산 엔진' 묶음. GPU엔 SM이 여러 개.
- **CUDA 코어:** SM 안의 실제 ALU. 실수 덧셈·곱셈 수행.
- **텐서 코어 (딥러닝의 핵심):** **타일 행렬곱(예: 16×16)을 한 명령으로** 처리하는 전용 하드웨어 연산기. LLM 행렬곱이 빠른 핵심 이유. *(큰 행렬은 이 타일 연산을 여러 번 누적 — "한 사이클에 전부"는 1차 근사)*
- **Register File:** 각 스레드 전용 초고속 메모리. 연산 중 중간값(가중치 갱신 델타 등)을 즉시 처리.
- **Shared Memory (스크래치패드):** SM 내부의 빠른 SRAM. CPU의 L1 캐시와 비슷하나 **개발자가 직접 Load/Store 제어**가 핵심 차이. 루프에서 반복 참조하는 행렬 조각을 여기 둠.
- **VRAM (Global Memory):** '비디오 메모리'. SM들이 공유해 읽고 쓰는 가장 큰 영토(모델 가중치·미니배치 적재). → [[메모리·캐시·멀티프로세스 (시스템 원리)|메모리 계층]]
- **버스(Memory Interface):** VRAM↔코어를 잇는 고대역폭 통로.

## 3. 🧠 ARM(SISD) → GPU(SIMT) 상태 전이 — 내 어셈블리 지식으로 장악
ARMv5의 폰 노이만/SISD 추론 모델을 GPU로 '상태 전이'하면 즉시 이해된다.

| ARMv5 (SISD) | GPU (SIMT) | 핵심 변화 |
|---|---|---|
| `PC` 이동 → Fetch·Decode·Execute 순차 | 스케줄러가 **Decode 1회 → 수천 코어 브로드캐스트** | 제어부를 칩에서 걷어내고 그 자리를 ALU로 채움 |
| 제어부 1 : 데이터 1 | 제어부 1 : 데이터 수천 | SISD → SIMT |
| `Code/Data Segment`(MMU([ɛm ɛm juː], 엠엠유, 엠엠유)·TLB([tiː ɛl biː], 티엘비, 티엘비)) | **커널 바이너리**를 VRAM 특수 영역에 로드, 스케줄러가 시작 주소 지목 | 세그먼트 → '커널 뱅크' |
| `R0–R15` 고정 레지스터 | 스레드마다 대량 레지스터 할당 | 고정 → 동적 |
| `LDR/STR/ADD/MUL` (1:1 하드웨어 동작) | **타일 행렬곱 매크로 명령**(Tensor Core Instr) | 스칼라 → 매크로(행렬) |
| `ADD`를 1,000번 반복 | 1 하드웨어 명령으로 행렬 전체(타일 누적) | 반복 → 병렬 |

> [!tip] 🪂 정확성 보강 — '스택·재귀 부재'는 1차 근사   `#사이드/심화`
> "GPU엔 스택이 없어 재귀 금지"는 *대체로 맞지만 엄밀히는 1차 근사*. 현대 GPU(CUDA Compute ≥2.0)는 스레드별 **로컬 메모리/스택이 있어 재귀·함수 호출도 가능**하다 — 다만 스레드 수천 개가 각자 깊은 스택을 쓰면 비용이 폭증해 **회피**한다. 그래서 GPU 커널은 재귀·깊은 호출 대신 **플랫(flat)한 반복문·배열 연산**으로 작성된다. → 우리가 보는 딥러닝 훈련 루프가 *단순 2중 루프*인 하드웨어적 이유.

## 4. 분기의 저주 (Branch Divergence) — '로직이 없다'의 진짜 의미
GPU가 복잡한 로직에 약한 결정적 이유.
- **CPU:** `if (x>0)`이 오면 분기 예측기가 다음 흐름을 미리 채워 둠(파이프라인 유지).
- **GPU(SIMT):** 수천 코어가 *동일 명령*을 동시에 수행. 1,000 코어 중 절반이 `if`, 절반이 `else`면 **두 그룹을 순차 실행**(= 분기 발산) → **절반이 노는** 상황, 효율 반토막.
- → 그래서 GPU 프로그래밍은 `if`를 피하고 모든 데이터를 행렬로 묶어 끝까지 연산하는 **분기 없는 코드(Branchless)** 를 강제한다.

## 5. 🌟 하드웨어가 알고리즘을 강제한다 (핵심 통찰)
딥러닝 학습(순전파+역전파)은 수학적으로 **행렬 곱(GEMM, `Y = WX + b`)** 으로 환원된다. 왜 하필?

- **행렬곱은 병렬성이 극대:** 결과 행렬의 각 셀은 *다른 셀의 결과가 필요 없다* → 수천 코어에 찢어 동시 계산 가능.
- 그래서 **"행렬 곱만 하면 GPU 가속이 극대화되도록 딥러닝 알고리즘을 설계"** 한 것. 인과가 거꾸로다 — *GPU가 빨라서 행렬곱을 쓰는 게 아니라*, **GPU가 잘하는(분기 없는 행렬곱) 형태로 딥러닝을 맞춘** 것.
- 즉 `NeuralNetwork`는 복잡한 제어 로직 알고리즘이 아니라 **"입력·가중치를 층층이 곱하고 더하는, 분기가 최소화된 초거대 정적 수치 회로"** 다. **하드웨어 제약(분기 없는 코드만)이 딥러닝 아키텍처(왜 계층적 행렬곱인지)를 강제**했다.

> [!quote] 🌉 타 영역 연결 — 하드웨어 ↔ 알고리즘 공생   `#연결/AI`
> "왜 딥러닝은 층층이 쌓인 행렬곱인가?"의 답이 여기 있다. 정렬: 정적 수치 회로(분기 없음) → [[정적 vs 동적 — 컴파일타임과 런타임이 가르는 것들|정적 vs 동적]]. 행렬곱의 선형대수 토대 → [[02 텐서 이해하기|텐서·matmul]]. 전역 다리 → [[_지식 연결 허브]]

### 🔢 왜 3D(고차원)도 2D(행렬)로 펴는가 — 동차좌표·텐서코어·stride([straɪd], 스트라이드, 스트라이드)
> 의문: *"게임 렌더링 등 3D 계산에 행렬식을 쓰는 건, GPU가 고차원 배열을 못 다뤄서 2D로 맞추는 건가?"* → **아니다.** ① 수학적 통일성 ② 텐서 코어(2D 타일) 물리 설계 ③ 메모리 stride 효율 때문에 *2D(행렬) 형식을 고수*하는 것. (고차원은 못 다루는 게 아니라 reshape해 2D GEMM으로 처리)

**① 수학적 통일성 — 동차좌표(Homogeneous Coordinates([həˈmɒdʒɪnəs koʊˈɔːrdɪnəts], 호모지니어스 코오디네이츠, 호모지니어스 코디네이츠))** _(수식·전개 전체 → [[3D 변환 행렬 — 동차좌표·회전·투영 (선형대수 응용)|수학 노트]])_
- 3D 변환(회전·이동·투영)은 전부 선형 변환 → 행렬 곱으로 표현. 단 *이동(translation([trænsˈleɪʃən], 트랜슬레이션, 트랜슬레이션))* 은 3×3 선형만으론 표현 불가 → 점 `(x,y,z)`를 **`(x,y,z,1)`로 확장하고 4×4 행렬**을 쓴다.
- 이점: 회전→이동→투영을 매번 따로 적용하지 않고, **변환 행렬들을 먼저 곱해(결합법칙) 단일 통합 행렬**을 만든 뒤 벡터에 한 번만 곱한다. = 복잡한 공간 변환을 *단일 연산(행렬곱)으로 표준화*.

**② 하드웨어 — 텐서 코어는 2D 타일 매크로**
- 텐서 코어 내부 회로가 **4×4·8×8 타일 행렬곱(MMA([ɛm ɛm eɪ], 엠엠에이, 엠엠에이))을 한 명령으로** 굴리도록 설계(§2·§3). ARM `MUL`이 정수 1개를 곱한다면, GPU `GEMM` 명령 1개는 **행렬 블록 전체**를 연산.
- 고차원 텐서를 *그대로* 텐서 코어에 못 꽂는다 → **reshape([riːˈʃeɪp], 리셰이프, 리셰이프)/flatten([ˈflætən], 플래튼, 플래튼)해 2D GEMM 형태로 펴서** 공급(im2col([ˈɪmtuːkɒl], 임투콜, 임투콜) 등). "고차원 불가"가 아니라 "2D로 펴야 가속 회로를 탄다".

**③ 메모리 — 규칙적 stride(지역성)**
- 2D 행렬(행 우선 평탄화)은 주소 계산이 **`y*width + x`** 로 단순 → ARMv5의 `Base + Index*Size`와 같은 패턴. 이 규칙적 **stride** 덕에 수천 코어가 VRAM에서 연속으로 읽어 텐서 코어에 끊김 없이 공급(파이프라인 유지). → [[메모리·캐시·멀티프로세스 (시스템 원리)|연속 메모리·캐시 히트]]
- 고차원을 그대로 쓰면 `(z,y,x)` 주소 로직이 복잡해져 하위(커널)에서 주소 계산 오버헤드 ↑. 2D면 그 주소 로직조차 하드웨어 회로로 고정.

> 한 줄: **"GPU는 고차원을 못 푸는 바보가 아니라, 행렬 곱이라는 고속도로를 깔고 그 위에서만 달리는 슈퍼카. 고차원 데이터도 코어에 넣을 땐 2D(행렬)/1D(평탄 벡터)로 정렬해야 그 고속도로를 탄다."** → §5 "하드웨어가 알고리즘을 강제"의 구체 사례. (행렬곱·reshape 선형대수 → [[02 텐서 이해하기|텐서·matmul]])

## 6. 개발자 매핑 — 모델이 GPU에서 도는 법 (정확판)
ARM 어셈블리 뷰로 치환하되, *노트 09/10의 '구조 vs 상태' 구분을 유지*한다.

1. **커널 로드:** 신경망 연산(행렬곱·ReLU([ˈrɛluː], 렐루, 렐루))의 GPU용 **커널 바이너리**가 *CUDA/Metal([ˈmɛtəl], 메탈, 메탈) 드라이버*에 의해 VRAM 명령어 영역에 로드(ARM의 ELF([ɛlf], 엘프, 엘프) 섹션 로드와 유사). *(이건 드라이버·JIT([dʒɪt], 짓, 지트)가 하는 일)*
2. **상태(가중치) 배치:** `model.to(device)` = 모델의 **가중치 텐서(state)를 VRAM Global Memory에 상주**시킴. ⚠️ *`model.to()`가 옮기는 건 가중치 텐서지 커널 바이너리가 아니다*(흔한 혼동 — [[10 GPU 가속과 분산 훈련 (DDP)|노트 10]] 참조). 구조(클래스 코드)는 CPU의 파이썬 객체로 남음.
3. **병렬 커널 호출:** `forward()` 시 CPU(장군)가 스케줄러에 `Kernel(VRAM_ptr_A, VRAM_ptr_B, ptr_C)` 명령(포인터)을 쏘고 결과만 기다린다 → CPU는 **연산을 직접 안 하고 '관리자'로 역할 전환**.
4. **병렬 실행:** 수천 스레드가 각자 VRAM의 다른 주소 offset을 레지스터로 로드, **동일 커널 로직을 각자 데이터에 적용**(병렬).

> [!example] 🧭 사이드 · 비유 — 훈련 루프 = ARM 시퀀스의 대규모 병렬판   `#사이드/비유`
> 딥러닝 훈련 루프를 ARM 어셈블리로 치면 **`[LDM(로드) → GEMM(행렬곱) → STM(저장) → ADD(가중치 갱신) → B(반복)]`** 시퀀스가 *수천 스레드에서 동시에* 도는 루틴. ARMv5가 `ADD` 1,000번 반복할 걸, GPU는 매크로 명령 1번으로 끝낸다. = "복잡한 로직을 스스로 고민"이 아니라 "이미 로드된 로직을 받아 일꾼 수천이 데이터를 찢어 병렬 처리하는 가속기".

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **SIMT** (Single Instruction, Multiple Threads) | 제어부 1개가 같은 명령을 수천 스레드에 브로드캐스트(GPU 실행 모델) |
| **SISD** (Single Instruction, Single Data) | 제어부 1 : 데이터 1의 순차 실행(ARM 등 폰 노이만 CPU) |
| **SM** (Streaming Multiprocessor) | GPU의 큰 연산 블록 = 코어·레지스터·Shared Memory 묶음 |
| **CUDA 코어** | SM 안의 단순 ALU(실수 덧셈·곱셈) |
| **텐서 코어** | 타일 행렬곱(MMA) 전용 하드웨어 연산기 |
| **Shared Memory** | SM 내부의 *개발자가 제어하는* 빠른 SRAM(스크래치패드) |
| **지연 vs 처리량** | Latency(한 작업 빨리, CPU) vs Throughput(많은 작업, GPU) |
| **분기 발산** (Branch Divergence) | SIMT에서 `if/else`로 스레드가 갈려 순차 실행되며 효율 저하 |
| **분기 없는 코드** (Branchless) | `if`를 피하고 행렬 연산으로 끝까지 — GPU 최적화 강제 |
| **GEMM** | General Matrix Multiply(`Y=WX+b`); 딥러닝 연산의 핵심 환원형 |
| **커널 (Kernel)** | GPU 코어들이 공유 실행하는 병렬 함수(드라이버가 VRAM에 로드) |
| **동차좌표** (Homogeneous Coord.) | 3D 점을 `(x,y,z,1)`로 확장 → 회전·이동·투영을 단일 4×4 행렬곱으로 통합 |
| **stride (스트라이드)** | 메모리에서 다음 원소까지 간격; 규칙적 stride라야 코어가 연속 읽어 가속 |
