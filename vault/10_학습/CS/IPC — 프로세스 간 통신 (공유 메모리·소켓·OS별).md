# IPC([aɪ piː siː], 아이피시, 아이피시) — 프로세스 간 통신 (공유 메모리 · 소켓 · OS([oʊ ɛs], 오에스, 오에스)별 구현)

> OS는 프로세스의 메모리를 철저히 격리한다. 그런데도 프로세스끼리 데이터를 주고받아야 할 때, **합법적으로 벽을 넘는 온갖 방법**을 통칭해 IPC(Inter-Process Communication([ˈɪntər ˈproʊsɛs kəˌmjuːnɪˈkeɪʃən], 인터 프로세스 커뮤니케이션, 인터 프로세스 커뮤니케이션)). 그 두 방식과 OS별 구현, 그리고 "왜 로컬 통신엔 TCP/IP([tiː siː piː aɪ piː], 티시피아이피, 티시피아이피)보다 IPC가 빠른가".

> 🧩 **CS 노트(원리·범용)** — OS·시스템 프로그래밍의 토대.
> 🔗 영역 간 연결: 🌐 [[_지식 연결 허브]] ｜ 📖 [[_CS MOC]] ｜ 🔗 토대: [[메모리·캐시·멀티프로세스 (시스템 원리)|메모리·멀티프로세스]]
> 🛠️ PyTorch 적용 → [[T03 PyTorch 데이터 로딩 성능 최적화|PyTorch 데이터 로딩]] (`num_workers`가 OS별로 다르게 동작)

> 🎯 **핵심 (TL;DR)**
> - **IPC = 격리된 프로세스가 데이터를 주고받는 방법 통칭.** 프로세스 메모리는 격리(보안)되므로 필요.
> - **두 방식:** ① **공유 메모리**(공용 폴더에 쓰고 읽기 — 빠름, 동기화 필요) ② **메시지 패싱**(파이프·큐·소켓 — 줄 서서 안전, 직렬화로 대용량 느림).
> - **로컬은 IPC가 TCP/IP보다 수배~수십배 빠르다** — 네트워크 스택(패킷화·체크섬·컨텍스트 스위칭)을 통째로 바이패스.
> - **OS별 구현이 다르다:** Linux([ˈlɪnəks], 리눅스, 리눅스) `/dev/shm`+fork([fɔːrk], 포크, 포크) · macOS([mæk oʊ ɛs], 맥오에스, 맥오에스) POSIX([ˈpɒzɪks], 포직스, 포식스) shm(용량 작음)+spawn([spɔːn], 스폰, 스폰) · Windows([ˈwɪndoʊz], 윈도우즈, 윈도우) Named Shared Memory([neɪmd ʃɛrd ˈmɛməri], 네임드 셰어드 메모리, 네임드 셰어드 메모리)+spawn.

## 1. IPC가 존재해야 하는 이유 — 프로세스 격리
- 실행 중인 프로그램(프로세스)들은 서로 남남이다. 크롬이 메모장의 메모리를 마음대로 읽으면 보안 재앙 → **OS가 프로세스 간 메모리 접근을 철저히 금지**(프로세스 격리).
- 하지만 프로세스끼리 데이터를 주고받아야 할 일은 늘 생긴다(예: 부모가 워커에게 "데이터 가져와"). 이 격리된 벽을 **합법적으로** 넘는 기술들의 총칭이 **IPC**.

## 2. IPC의 두 가지 대표 방식
**① 공유 메모리 (Shared Memory([ʃɛrd ˈmɛməri], 셰어드 메모리, 셰어드 메모리)) = "공용 폴더" 방식**
- OS가 여러 프로세스가 동시 접근 가능한 특수 메모리 구역을 연다. A가 거기 쓰면 B가 그냥 읽어간다.
- **빠름**(복사 오버헤드 없음 — 포인터만). 단점: 여러 명이 한 폴더를 동시 수정하면 꼬이므로 **동기화(Lock 등)를 직접 관리**해야 함.
- *PyTorch `DataLoader`가 쓰는 방식.*

**② 메시지 패싱 (Message Passing([ˈmɛsɪdʒ ˈpæsɪŋ], 메시지 패싱, 메시지 패싱)) = 파이프·큐·소켓 = "전화/카톡" 방식**
- 공용 공간이 아니라, OS가 두 프로세스 사이에 **빨대(Pipe([paɪp], 파이프, 파이프))**를 꽂는다. A가 밀어 넣으면 B가 순서대로 받아먹음.
- **안전**(줄 서서 받아 동기화 문제 없음). 인터넷 통신(Socket([ˈsɒkɪt], 소킷, 소켓))도 이 계열. 단점: 대용량 데이터는 빨대 규격에 맞게 **잘게 쪼갰다 합치는 직렬화**가 필요해 느림.

## 3. OS별 구현 차이 (개발자 관점)
같은 '공유 폴더'라도 하드웨어·커널 철학이 OS마다 다르다.

| 항목 | Linux | macOS | Windows |
|---|---|---|---|
| 기반 | 유닉스 계열(Open Source) | 유닉스 계열(BSD/Darwin) | Windows NT 커널 |
| 프로세스 생성 | **`fork`** (빠름, 메모리 복사) | **`spawn`** (안정성 위해 강제) | **`spawn`** (구조상 필수) |
| 공유 메모리 | **`/dev/shm`** (RAM 위 가상 폴더) | POSIX `shm_open`+`mmap`, 단 **용량 작음** → 파일 시스템 우회 | **Named Shared Memory** (`CreateFileMapping`/`MapViewOfFile`, 페이징 파일 기반) |
| 특징·한계 | 대용량 병렬에 가장 최적 | 커널 공유 메모리 **용량 제한 엄격** | 프로세스 생성 비용 큼·직렬화 오버헤드 |

- **Linux:** RAM을 폴더처럼 쓰는 `/dev/shm`에 텐서를 올려 빨대 없이 공유. fork로 부모 메모리를 복사해 빠르게 자식 생성.
- **macOS:** POSIX(`shm_open`·`mmap`)를 쓰지만 커널 공유 메모리 최대치(`shmmax`·`shmall`)가 작아, 대용량을 올리면 `RuntimeError: received 0 items of ancdata` 류 에러 → **임시 파일을 만들어 `mmap`하는 파일 시스템 전략**으로 우회. 또 CoreFoundation([ˌkɔːr faʊnˈdeɪʃən], 코어 파운데이션, 코어 파운데이션) 충돌 방지로 **`spawn` 강제**.
- **Windows:** `fork`가 없어 **완전히 새 인터프리터를 띄우는 `spawn`**. 공유 메모리는 페이징 파일에 가상 구역을 할당하고 `"PyTorch_Tensor_XYZ"` 같은 **고유 이름(Named)** 으로 접근 — 경로 대신 *이름*이 공용 폴더의 열쇠.

## 4. 왜 로컬 통신엔 TCP/IP보다 IPC인가
같은 컴퓨터 안 프로세스끼리도 익숙하다고 TCP/IP 소켓(루프백 `127.0.0.1`)을 쓰면 하드웨어 레벨에서 낭비가 크다.

**TCP/IP 루프백의 오버헤드:**
- 데이터를 잘게 쪼개 TCP/IP 헤더 붙이기(패킷화), 흐름·혼잡 제어 계산, 체크섬 검증, 커널↔유저 공간을 넘나드는 복사·컨텍스트 스위칭.
- 물리 랜선만 안 나갈 뿐, **네트워크 카드 드라이버 직전까지의 거대한 복잡도**를 그대로 감당 → CPU·캐시를 갉아먹음.

**로컬 IPC(공유 메모리·유닉스 도메인 소켓 UDS([juː diː ɛs], 유디에스, 유디에스))의 이점:**
- **네트워크 스택을 통째로 바이패스.** IP·포트·패킷 개념이 개입하지 않음.
- 자식이 공유 메모리에 쓰면 메인은 그 주소를 그냥 읽고 끝. **CPU 캐시 히트가 극대화** → TCP/IP 루프백보다 수배~수십배 빠름.

## 5. 역사와 현대 — 왜 'IPC 채팅 프로그램'이 교과서 단골인가
**역사(메인프레임·유닉스 시대):** 과거엔 거대한 컴퓨터(서버) 한 대에 멍청한 단말기(Dumb Terminal([dʌm ˈtɜːrmɪnəl], 덤 터미널, 덤 터미널)) 수십 대가 붙어 동시 로그인했다. "A와 B가 채팅" = 서로 다른 컴퓨터가 아니라 **하나의 OS 커널 안에서 도는 A의 프로세스와 B의 프로세스가 대화**. 외부로 나갈 필요가 없어 네트워크보다 **로컬 IPC(파이프·메시지 큐)** 로 채팅을 만드는 게 당연했다.

**현대에도 여전히:**
- **DB([diː biː], 디비, 디비) 고속 통신:** 웹 서버와 DB(MySQL([maɪ ˌɛs kjuː ˈɛl], 마이에스큐엘, 마이에스큐엘)·Postgres([ˈpoʊstɡrɛs], 포스트그레스, 포스트그레스))가 같은 서버 안이면 TCP 포트(`3306` 등) 대신 **유닉스 도메인 소켓**으로 통신 → 처리량 20~30%↑.
- **마이크로서비스/Docker([ˈdɒkər], 도커, 도커):** 한 머신의 컨테이너([kənˈteɪnər], 컨테이너, 컨테이너)끼리 대용량을 빠르게 주고받을 때, 가상 네트워크(veth([viː ɛθ], 브이이에스, 브이이에스)) 병목을 피해 **공유 메모리 IPC 영역을 공유**.

> 결론: IPC 채팅 예제가 단골인 건 *구조가 단순해서*가 아니라, **"같은 하드웨어 안에서 성능을 극한으로 쥐어짜는 가장 올바른 엔지니어링 방법"** 이라서.

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **IPC** (Inter-Process Communication) | 격리된 프로세스가 데이터를 주고받는 방법 통칭 |
| **공유 메모리** (Shared Memory) | 여러 프로세스가 접근하는 공용 메모리 구역(복사 없이 빠름, 동기화 필요) |
| **메시지 패싱** (Message Passing) | 파이프·큐·소켓으로 줄 서서 주고받기(안전, 직렬화로 대용량 느림) |
| **파이프 / 소켓** (Pipe / Socket) | 프로세스 사이 데이터 통로. 소켓은 네트워크 통신에도 사용 |
| **유닉스 도메인 소켓** (UDS) | 같은 머신 내 프로세스 간 소켓(네트워크 스택 없이 로컬) |
| **`/dev/shm`** | Linux의 RAM 위 가상 공유 폴더 |
| **Named Shared Memory** | Windows의 이름 기반 공유 메모리(`CreateFileMapping`) |
| **fork vs spawn** | 부모 메모리 복사(빠름) vs 새 인터프리터 기동(안정·느림) |
| **TCP/IP 루프백** (`127.0.0.1`) | 같은 머신 내 TCP 통신; 네트워크 스택 오버헤드 발생 |
| **직렬화** (Serialization) | 데이터를 전송용 바이트열로 쪼갰다 복원하는 과정 |
| **컨텍스트 스위칭** | 커널↔유저/프로세스 전환 비용 |
