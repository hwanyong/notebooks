## 1. 메모리 계층과 CPU 캐시
- 저장장치는 **계층 구조**다: 디스크(SSD) ≪ RAM(주기억) ≪ CPU 캐시(L3<L2<L1) ≪ 레지스터. 위로 갈수록 **빠르지만 작고 비싸다.**
- **캐시 히트/미스:** CPU가 원하는 데이터가 캐시에 있으면 히트(빠름), 없으면 미스 → 느린 RAM/디스크까지 내려가 가져옴. **히트율이 성능을 좌우.**
- **캐시 라인(Cache Line([kæʃ laɪn], 캐시 라인, 캐시 라인)):** CPU는 RAM([ræm], 램, 램)에서 1바이트가 아니라 **덩어리(보통 64바이트) 단위**로 가져온다. 그래서 "지금 쓰는 것 옆에 다음에 쓸 게 붙어 있으면" 공짜로 따라와 빨라진다.

## 2. 캐시 히트를 높이는 메모리 배치 원리
**① 연속 메모리(C-contiguous) vs 포인터 추적(Pointer Chasing([ˈpɔɪntər ˈtʃeɪsɪŋ], 포인터 체이싱, 포인터 체이싱))**
- 데이터가 메모리에 **끊김 없이 연속**으로 놓이면, 순차 접근 시 CPU가 인접 데이터를 캐시 라인에 미리 끌어와 히트율이 치솟는다.
- 반대로 **객체들의 포인터 주소만 모아 둔 구조**(예: 동적 언어의 리스트)는 인덱싱마다 메모리 사방을 점프(Pointer Chasing) → 캐시 라인이 통째로 무효화되고 매번 RAM까지 다녀온다.

**② 데이터 타입 크기 ↔ 캐시 라인 밀도**
- 64바이트 캐시 라인에 8바이트 타입은 8개, 1바이트 타입은 64개가 들어간다. **타입을 작게** 쓰면 라인당 데이터 밀도가 올라 **캐시 미스 확률이 수학적으로 떨어진다.** (정밀도가 충분하다면 작은 타입이 유리)

**③ 지역성(Locality([loʊˈkælɪti], 로컬리티, 로컬리티))**
- **공간 지역성:** 곧 쓸 데이터가 *주소상 인접*할수록 유리(연속 배열).
- **시간 지역성:** 방금 쓴 데이터를 *곧 다시* 쓰면 캐시에 남아 있어 유리.

**④ 벡터화(Vectorized Operation([ˈvɛktəraɪzd ˌɒpəˈreɪʃən], 벡터라이즈드 오퍼레이션, 벡터라이즈드 오퍼레이션))**
- 한 원소씩 도는 인터프리터 루프 대신, **연속 배열을 통째로 처리하는 저수준(C/SIMD([sɪmd], 심드, 심드)) 연산**으로 끝내면 캐시·파이프라인 효율이 극대화된다. (반복을 '행렬 연산'으로 표현하는 것과 같은 맥락)

## 3. 멀티코어의 함정 — False Sharing
- 여러 코어가 각자 캐시를 갖는다. 한 코어가 **같은 캐시 라인에 있는 (논리적으로 무관한) 옆 데이터**를 수정하면, 하드웨어는 정합성을 위해 **다른 코어의 그 캐시 라인까지 무효화**한다 = **False Sharing**.
- 결과: 서로 다른 데이터를 다루는데도 캐시가 계속 깨져 **캐시 일관성(Cache Coherency([kæʃ koʊˈhɪərənsi], 캐시 코히어런시, 캐시 코히어런시)) 유지 비용**이 폭발.
- 원리적 회피: 코어/스레드마다 **독립된(겹치지 않는) 메모리 영역**에서 작업하고, 공유 데이터는 **읽기 전용**으로 둔다.

## 4. 멀티프로세스의 메모리 구조
**① 멀티스레드 vs 멀티프로세스**
- **멀티스레드:** 한 프로세스의 **heap([hiːp], 힙, 힙)을 공유**하는 여러 실행 흐름. 메모리 공유가 쉽지만, 언어에 따라 한계가 있다.
- **GIL(Global Interpreter Lock([ˈɡloʊbəl ɪnˈtɜːrprɪtər lɒk], 글로벌 인터프리터 락, 글로벌 인터프리터 록)):** 일부 언어(파이썬)는 인터프리터 락 때문에 한 heap을 공유하는 스레드로는 *진짜 병렬 연산이 불가능*. 그래서 병렬이 필요하면 **독립 프로세스를 여러 개** 띄운다(멀티프로세스).
- **프로세스 메모리 격리:** OS는 프로세스끼리 서로의 메모리(stack([stæk], 스택, 스택)·heap 모두)를 **못 보게 격리**한다. 한 프로세스는 다른 프로세스의 heap에 직접 접근 불가.

**② heap / stack**
- **heap:** 동적 할당 영역(객체·배열). 크고 오래 산다.
- **stack:** 함수 호출의 지역 변수 영역. 함수가 끝나면 사라지고, **그 함수/스레드에 사적**이다.

**③ 프로세스 간 통신 — IPC**
- 서로의 heap을 못 보므로, **OS가 제공하는 통신 수단(IPC)** 으로 주고받는다. 대표는 **공유 메모리**(공용 구역에 복사 없이 포인터로 — 빠름)와 **메시지 패싱**(파이프·소켓 — 안전하나 직렬화). Linux는 RAM 위 `/dev/shm`.
- → 두 방식·OS별 구현(Linux/macOS/Windows)·TCP/IP 루프백 비교·역사는 독립 노트: [[IPC — 프로세스 간 통신 (공유 메모리·소켓·OS별)|IPC 노트]].

**④ OS 페이지 캐시 (디스크 ↔ RAM)**
- OS는 디스크에서 한 번 읽은 파일을 버리지 않고 **RAM 빈 공간(Page Cache([peɪdʒ kæʃ], 페이지 캐시, 페이지 캐시))** 에 캐싱한다. 같은 파일을 다시 읽으면 디스크를 안 건드리고 RAM에서 복사 → 수배 빠르다.

**⑤ DMA([diː ɛm eɪ], 디엠에이, 디엠에이) (RAM ↔ 장치)**
- **Direct Memory Access([dɪˈrɛkt ˈmɛməri ˈæksɛs], 다이렉트 메모리 액세스, 다이렉트 메모리 액세스):** CPU가 일일이 나르지 않고, 장치(GPU 등)가 RAM과 **직접** 데이터를 전송하는 기법. 단, 대상 메모리가 물리 주소에 **고정(page-lock)** 돼 있어야 가능. CPU를 풀어줘 비동기 고속 전송이 된다.

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **메모리 계층** | 디스크≪RAM≪캐시≪레지스터; 위로 갈수록 빠르고 작음 |
| **캐시 히트/미스** | 원하는 데이터가 캐시에 있음/없음. 히트율이 성능 좌우 |
| **캐시 라인** (Cache Line) | CPU가 RAM에서 한 번에 가져오는 최소 단위(보통 64바이트) |
| **C-contiguous** | 데이터가 메모리에 끊김 없이 연속 배치된 상태 |
| **Pointer Chasing** | 포인터를 따라 메모리 사방을 점프 → 캐시 무효화 |
| **공간/시간 지역성** (Locality) | 곧 쓸 게 주소상 가깝다/방금 쓴 걸 곧 다시 쓴다 |
| **벡터화** (Vectorized Op) | 인터프리터 루프 대신 연속 배열을 저수준(C/SIMD)으로 일괄 처리 |
| **False Sharing** | 다른 코어가 같은 캐시 라인의 인접 데이터를 수정해 내 캐시가 무효화 |
| **캐시 일관성** (Cache Coherency) | 여러 코어 캐시 간 정합성을 맞추는 비용 |
| **멀티스레드 vs 멀티프로세스** | heap 공유 스레드 vs 메모리 격리된 독립 프로세스 |
| **GIL** | 전역 인터프리터 락; 한 heap 공유 스레드의 병렬을 막음 |
| **프로세스 격리** | OS가 프로세스끼리 메모리를 못 보게 막는 것 |
| **heap / stack** | 동적 할당 영역(heap) vs 함수 지역 변수 영역(stack) |
| **IPC / 공유 메모리** | 프로세스 간 통신용 특수 공유 영역(예: `/dev/shm`) |
| **페이지 캐시** (Page Cache) | OS가 디스크에서 읽은 파일을 RAM 빈 공간에 캐싱 |
| **DMA** | CPU 개입 없이 RAM↔장치(GPU) 직접 전송(메모리 고정 필요) |
