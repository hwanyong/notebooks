# 정적 vs 동적 — 컴파일타임과 런타임이 가르는 것들

> **CS([siː ɛs], 시에스, 시에스)의 한 잣대가 여러 영역을 가른다:** "지식·데이터가 *사전에 고정*(컴파일타임)인가, *실행 중 실시간 결정*(런타임)인가." 이 하나의 기준이 코드 구조·ML 학습 방식(지도/강화)·최신 추론 모델(o1)까지 똑같이 가른다.

> 🌐 영역 간 연결: [[_지식 연결 허브]] ｜ 🔙 [[_통찰 MOC]]
> 🧪 발단: "딥러닝 코드와 강화학습 코드가 왜 이렇게 다르게 생겼나"를 *정적/동적 = 컴파일타임/런타임* 으로 직관한 것.

> 🎯 **핵심 (TL;DR)**
> - **정적(컴파일타임):** 실행 전에 모든 게 고정. 컴파일러가 최적화하기 좋은 "거대한 행렬 연산 덩어리".
> - **동적(런타임):** 실행 흐름 속에서 실시간 생성·지연 결정(Late Binding([leɪt ˈbaɪndɪŋ], 레이트 바인딩, 레이트 바인딩)). 조건 분기·상태 변화 추적하는 "시뮬레이션 루프".
> - **ML([ɛm ɛl], 엠엘, 엠엘) 대응:** 지도학습/LLM([ɛl ɛl ɛm], 엘엘엠, 엘엘엠) = 정적, 강화학습 = 동적.
> - **o1([oʊ wʌn], 오원, 오원)/o3([oʊ θriː], 오쓰리, 오쓰리) = 정적 모델 + 동적 루프의 '링킹'** (컴파일된 LLM 뼈대 위에 런타임 RL([ɑːr ɛl], 알엘, 알엘) 탐색 루프를 연결).

---

## 1. 한 잣대 — '사전 고정' vs '실행 중 결정'
전통 CS에서 정적/동적을 가르는 절대 기준: **컴파일타임에 박히는가, 런타임에 결정되는가.**
- **정적(Static([ˈstætɪk], 스태틱, 스태틱)):** 빌드 시점에 값·구조·연결이 확정. 빠르고 예측 가능, 대신 경직.
- **동적(Dynamic([daɪˈnæmɪk], 다이내믹, 다이내믹)):** 실행 중에 결정(Late Binding). 유연하고 상황 적응, 대신 예측 불가·오버헤드.

## 2. CS의 세 가지 정적/동적 대비 (토대)
| 축 | 정적 | 동적 |
|---|---|---|
| **페이지** | `index.html` (고정 렌더) | React([riˈækt], 리액트, 리액트)/SSR([ɛs ɛs ɑːr], 에스에스알, 에스에스알) (Action([ˈækʃən], 액션, 액션)에 따라 런타임 생성) |
| **메모리** | 스택(stack([stæk], 스택, 스택), 컴파일타임 크기 확정) | 힙(heap([hiːp], 히프, 힙), `malloc`·런타임 할당·포인터 연산) |
| **라이브러리** | `.a`/`.lib` (소스에 통째 embed([ɪmˈbɛd], 임베드, 임베드)) | `.so`/`.dll` (런타임 링킹·IPC([aɪ piː siː], 아이피시, 아이피시)) |
→ 공통: 정적은 *미리 다 정해 굳힌 것*, 동적은 *실행하며 그때그때 잡는 것*. (메모리 stack/heap·IPC → [[메모리·캐시·멀티프로세스 (시스템 원리)|CS 시스템 원리]])

## 3. ML 패러다임이 이 잣대를 그대로 따른다
**지도학습 / LLM = 정적 (컴파일타임)**
- 데이터: 디스크에 고정된 **불변(Immutable([ɪˈmjuːtəbəl], 이뮤터블, 이뮤터블)) 데이터셋**.
- 연산: 런타임 분기 없이 *사전 정의된 텐서 shape*(`[Batch, Seq, Hidden]`)로 거대 행렬곱을 일괄 처리.
- 결과 단방향: 출력이 학습 데이터를 바꾸지 못함.
- → [[08 훈련 루프 (Training Loop)|훈련 루프]]가 그 정적 대표(고정 데이터 → loss → backward).

**강화학습 = 동적 (런타임)**
- 고정 데이터셋 없음. 에이전트가 런타임에 `action`을 고르면 `next_state`·`reward`가 **메모리에 실시간 생성**(Late Binding).
- `env`와 메시지를 주고받으며(IPC/포인터 연산처럼) 점수판(`Q-table`/`V`)을 런타임에 mutable([ˈmjuːtəbəl], 뮤터블, 뮤터블) 갱신.
- 불확실성: 같은 상태·행동도 결과가 달라질 수 있는 완전 런타임 시뮬레이션.
- → 개념·코드는 [[강화학습 패러다임 — 정적 데이터 vs 실시간 상호작용|강화학습 패러다임]].

## 4. o1/o3 = 정적 + 동적의 '링킹'
- 전통 개발에서 정적 라이브러리를 동적 프로그램에 *로드*해 쓰듯,
- **사전에 컴파일(학습)된 정적 LLM(Transformer([trænsˈfɔːrmər], 트랜스포머, 트랜스포머) 뼈대)을 엔진으로 삼고, 그 위에 런타임에 정답 경로를 동적으로 탐색하는 RL 루프를 링킹**한 것이 o1/o3 같은 추론 모델의 본질.
- → [[LLM은 왜 길어질수록 평이해지나 — 어텐션 희석과 엔트로피|LLM 노트]]의 "추론 모델이 System 2를 강제"가 바로 이 동적 링킹.

> [!tip] 🪂 사이드 · 심화 — 왜 정적이 '컴파일러 친화'인가   `#사이드/심화`
> 정적이면 컴파일러·GPU가 *모든 걸 미리 알아* 최적화(메모리 사전 할당·병렬 행렬곱)가 쉽다. 동적이면 런타임에 분기·할당이 생겨 최적화가 어렵지만 유연하다. ML도 똑같다: 지도학습은 GPU 풀가동에 최적, 강화학습은 느려도 환경 적응.

## 🆕 새 용어 정리
| 용어 | 한 줄 정의 |
|---|---|
| **정적 / 동적** | 컴파일타임 고정 vs 런타임 결정 |
| **컴파일타임 / 런타임** | 빌드 시점 vs 실행 시점 |
| **Late Binding** | 값·연결을 실행 중에 늦게 결정 |
| **스택 / 힙** | 컴파일타임 크기 확정 vs 런타임 동적 할당 |
| **정적/동적 라이브러리** | `.a`(embed) vs `.so`(런타임 링킹) |
| **링킹** (Linking([ˈlɪŋkɪŋ], 링킹, 링킹)) | 정적 모듈을 동적 실행에 연결 — o1 = LLM+RL 링킹 |
